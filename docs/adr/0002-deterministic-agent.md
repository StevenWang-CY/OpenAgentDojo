# ADR 0002: Deterministic Agent (Hybrid Simulation)

- Status: Accepted
- Date: 2026-05-21
- Deciders: AgentSupervisor Arena team

## Context

The product grades supervision quality. To grade fairly, the agent's behavior must be repeatable: two users facing "Mission 01" must see the same flawed patch, and the same user replaying the mission must see the same patch on attempt 2. A pure-LLM agent fails every part of that requirement:

- **Non-deterministic patches** would make scoring incomparable across users.
- **Per-call cost** scales with concurrency; even cheap models add real spend at workshop volumes.
- **Replay** for the determinism test (§19.2) becomes impossible.
- **Failure mode integrity** can't be guaranteed — an LLM might "fix" the bug correctly by accident, hollowing out the scenario.

## Decision

Adopt a **hybrid simulation** model:

1. The substantive agent action — the diff applied to the workspace — is a pre-written `agent_patch.diff` checked into `missions/<id>/`. The agent service reads it, applies it, and emits a `patch.applied` event. The LLM is **never** consulted on this path.
2. The prose ("here's what I'm doing and why") is generated by rendering a Jinja2 template (`prompts/response.md`) seeded with the user's prompt summary, selected context, and the mission's failure-mode title. This is also deterministic.
3. Optionally — gated behind `features.llm_narration_enabled` — a Claude Haiku call rewrites the rendered seed for tone only, with strict length and banned-token validation. On any deviation, the seed is used verbatim.
4. **Grading never touches an LLM.** All scoring is pure-functional over the event log, the diff, the test results, and validator outputs.

## Consequences

### Positive

- Replays are byte-identical.
- Mission self-tests (§19.1) can assert exact score envelopes for "unmodified agent patch" and "ideal solution" submissions.
- Cost is bounded and predictable — even with narration on, we ship a constant number of cached tokens per scenario.
- The failure mode is guaranteed to fire every time, which keeps the pedagogical promise.

### Negative

- Less "magical" feeling than a real LLM agent — the dialogue is templated.
- Authoring effort moves from prompt engineering to scenario engineering: every mission needs a believable diff.
- The intent classifier (`prompts/intents.yaml`) needs to be maintained per scenario.

### Neutral

- Multi-turn behavior is limited to a small set of stocked responses (`agent_patch_addtest.diff`, `agent_patch_revision.diff`); rich open-ended dialogue is out of scope for MVP.

## Alternatives considered

- **Pure LLM agent with system prompt forcing a specific bug.** Tried in a prototype; the model regularly drifted, sometimes generated correct fixes, and made scoring incomparable. Rejected.
- **LLM agent with output validation + rejection sampling.** Possible, but cost/latency unpredictable and we still couldn't guarantee a stable diff.
- **LLM agent for prose + handler-applied diff (current design).** Adopted.
- **LLM agent everywhere including grading.** Considered for "prompt quality" rubric only; rejected for MVP because it makes determinism tests impossible. Will revisit as a post-MVP enhancement with a deterministic-rubric fallback.
